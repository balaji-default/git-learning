Git Commands Guide

Git Configuration

git config --global user.name "Your GitHub Username"

git config --global user.email "your-email@example.com"

These commands set up your Git global username and email, used for commit tracking.

Viewing Global Configuration

git config --global user.name - Displays your GitHub username

git config --global user.email - Displays your GitHub email

git config --global --list - Displays both username and email

Local Configuration

git config user.name "Local Username"

git config user.email "local-email@example.com"

If no scope (--global or --local) is mentioned, Git applies changes to the local repository by default.

Viewing Local Configuration

git config --local --list - Lists the local username and email settings

Git Clone Command

Clones a repository from a remote server (e.g., GitHub) to your local machine.

Viewing Hidden Git Files

To view hidden files, including the .git directory:

  • ls -a (Linux/macOS)
  • Get-ChildItem -Force (Windows PowerShell)

Git Status Command

Displays the current state of your working directory and staging area.

  • Untracked (U): New file, not yet staged.
  • Modified (M): File has been changed but not staged.
  • Staged (A): File is added to the staging area, ready for commit.
  • Committed: File is stored in the repository.

Git Add and Commit

git add filename - Adds a specific file to the staging area.

git add . - Adds all modified and untracked files in the directory.

git commit -m "Meaningful commit message"

Commits the staged changes with a descriptive message.

Good commit messages should be:

  • Concise and meaningful
  • Descriptive of the changes made
  • Consistent with a proper format

Git Push Command

git push origin main

  • Push: Sends committed changes to a remote repository.
  • Origin: The default name for the remote repository.
  • Main: The primary branch where changes are pushed.

git init

Description: Initializes a new Git repository in the local directory.

Usage:

git init

This creates a .git folder, making the directory a Git repository.

git remote add origin <repo-link>

Description: Links your local repository to a remote repository.

Usage:

git remote add origin https://github.com/username/repo.git

Now, origin is the alias for your remote repository.

git remote -v

Description: Displays the remote repositories linked to your local repo.

Usage:

git remote -v

Example output:

origin https://github.com/username/repo.git (fetch)
 origin https://github.com/username/repo.git (push)

git branch

Description: Lists all local branches.

Usage:

git branch

Example output:

* main
    feature-branch

The * indicates the currently active branch.

git branch -M <new-name> vs. git branch -m <new-name>

  • git branch -M main ‚Üí Forces renaming the branch to main, even if the new name already exists.
  • git branch -m main ‚Üí Renames the branch to main only if there is no conflict.

git branch -a

Description: Lists both local and remote branches.

Usage:

git branch -a

Example output:

* main 
      feature-branch  

      remotes/origin/main  

      remotes/origin/feature-branch

git branch -M main

Description: Renames the current branch to main.

Usage:

git branch -M main

This is commonly used to rename master to main.

git push origin main

Description: Pushes the main branch to the remote repository.

Usage:

git push origin main

This updates the remote repository with the latest commits from main.

git push -u origin main

Description: Pushes main to the remote and sets it as the upstream branch.

Usage:

git push -u origin main

After this, you can just use:

git push

instead of specifying origin main every time.

Creating a New Branch

1Ô∏è‚É£ Using git branch (Creates a branch but doesn‚Äôt switch)

Description: This command creates a new branch, but you will still be on the current branch until you switch manually.

This creates a new branch named feature1, but you will remain on the current branch.

2Ô∏è‚É£ Using git checkout -b (Creates and switches to the new branch)

Description: This command creates a new branch and immediately switches to it.

Usage:

This creates feature2 and switches to it immediately.

Description: This is the preferred method in newer versions of Git for creating and switching branches.

Usage:

This creates feature3 and switches to it.

Usage:

Running this command will show which branch you are currently on. If you created the branches using git branch without switching, you will still be on the main branch.

Fetching and Merging Changes from Main

When working in a feature branch, it's important to stay up to date with the latest changes from the main branch. This prevents conflicts and ensures your branch has the most recent code before pushing your changes.

1Ô∏è‚É£ git fetch origin main

Description: Fetches the latest changes from the remote main branch but does NOT apply them yet.

Usage:

git fetch origin main

This downloads the latest updates from the remote repository without modifying your current branch.

Example Output:

          From github.com:user/repo  

           * branch  main   -> FETCH_HEAD

At this stage, your local branch does not yet contain the updates from main. You need to merge them.

2Ô∏è‚É£ git merge main

Description: Merges the fetched updates from main into your current branch.

Usage:

git merge main

If there are no conflicts, the changes are merged automatically. If there are conflicts, Git will ask you to resolve them.

üîπ Summary of Fetch vs Merge

Command Description
git fetch origin main Downloads the latest changes from remote main but does not merge them.
git merge main Merges the fetched changes into your current branch.

üîπ When to Use git fetch Instead of git pull?

  • Use git fetch origin main when you want to see the updates first before merging.
  • Use git pull origin main if you want to fetch + merge in one step (but with less control).

Final Thoughts:
‚úÖ git fetch lets you review changes before merging.
‚úÖ git merge applies the changes to your branch.
‚úÖ This approach avoids unexpected conflicts when working with multiple team members.

3Ô∏è‚É£ git pull origin main

Description: A shortcut that does both fetch + merge in one step.

Usage:

git pull origin main

This command automatically downloads the latest changes and merges them into your current branch.

When to use it?

  • Use git pull when you're sure you want to merge the latest changes immediately.
  • If you want more control, use git fetch followed by git merge.

4Ô∏è‚É£ git rebase main

Description: Instead of merging, rebasing applies your branch's changes on top of the latest main.

Usage:

git rebase main

This creates a cleaner history by avoiding unnecessary merge commits. If there are conflicts, Git will pause and let you resolve them one by one.

üîπ Summary of Fetch, Merge, Pull, and Rebase

Command Description
git fetch origin main Downloads the latest changes from remote main but does not merge them.
git merge main Merges the fetched changes into your current branch.
git pull origin main Fetches and merges in one step.
git rebase main Applies your changes on top of the latest main, keeping history cleaner.

üîπ When to Use Each Command?

  • Use git fetch when you want to review changes before merging.
  • Use git merge when you want to bring main's changes into your branch.
  • Use git pull when you need to fetch + merge quickly.
  • Use git rebase when you want a clean commit history.

Final Thoughts:
‚úÖ git fetch lets you review changes before merging.
‚úÖ git merge integrates changes while preserving commit history.
‚úÖ git pull is a faster way to fetch and merge together.
‚úÖ git rebase keeps history linear and avoids extra merge commits.

üìå Pull Request (PR) in GitHub

Description: A Pull Request (PR) is used to merge changes from a feature branch into the main branch after review. It is commonly used when multiple developers work on different features and want to integrate them safely.

1Ô∏è‚É£ Pushing a Feature Branch to Remote

Before creating a pull request, ensure your changes are pushed to GitHub.

Usage:

git push origin feature1

This command pushes your local feature1 branch to GitHub, making it available for review and merging.

After pushing, GitHub may indicate:

"This branch is X commits ahead of main."

This means your branch has changes that are not yet merged into main. You will also see a "Contribute" button, which provides options like:

  • "Compare & pull request" ‚Üí Create a pull request for merging.
  • "Sync changes" (if applicable).

2Ô∏è‚É£ Creating a Pull Request on GitHub

After pushing your branch, follow these steps to create a PR:

  • Go to your repository on GitHub.
  • Click on the "Compare & pull request" button next to your branch.
  • Review the changes and add a meaningful title and description.
  • Click "Create Pull Request" to submit it for review.

3Ô∏è‚É£ Reviewing and Merging the Pull Request

- Once the PR is created, other team members or maintainers can review the changes.
- If needed, they may request modifications before merging.
- Once approved, click "Merge Pull Request" to merge the feature branch into the main branch.

4Ô∏è‚É£ Checking Status Locally

You can check if your branch is ahead of main using:

git status

5Ô∏è‚É£ Deleting the Merged Branch

After merging, the feature branch is no longer needed and can be deleted: 

git branch -d feature1

6Ô∏è‚É£ Updating the Local Repository

If you were working on a feature branch, switch back to the main branch and update it:

git checkout main
git pull origin main

üîÑ Deleting Remote Branches & Refreshing Remote Tracking

üöÄ Deleting a Local Branch

Description: Deletes a branch from your local machine.

Usage:

git branch -d feature3

If the branch has unmerged changes, force delete with:

git branch -D feature3

üåç Deleting a Remote Branch

Description: Removes a branch from the remote repository.

Usage:

git push origin --delete feature3

‚ùì Why Does remotes/origin/feature3 Still Show Up?

Even after deleting the remote branch, Git may still show it in the remote-tracking list when you run:

git branch -a

This happens because Git caches remote branches locally.

üîÑ Refreshing Remote Branch List

Description: Updates the remote-tracking branches and removes stale branches.

Usage:

git fetch -p

The -p or --prune option removes references to branches that no longer exist on the remote.

‚úÖ Final Check

After pruning, run the following to verify that remotes/origin/feature3 is gone:

git branch -a
